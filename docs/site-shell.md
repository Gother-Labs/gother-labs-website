# Shared Site Shell Maintenance

This site is intentionally static. GitHub Pages serves the committed HTML, CSS, JavaScript, and assets directly, without a framework or build step. The shared site shell still needs discipline because the same chrome appears in hand-authored pages and in HTML generated by `tools/sync-results.mjs`.

## Current Inventory

The shared shell includes:

- Header and primary navigation.
- Home wordmark and animated SVG symbol.
- Footer.
- Inter font preload.
- Favicon.
- Shared `styles.css` and `scripts.js` includes.
- Shared shell cache-busting version, currently `nav-wordmark-v1`.
- Canonical URL, Open Graph metadata, Twitter card metadata, and page-specific metadata exceptions.

The hand-authored shell currently appears in:

- `index.html`
- `company/index.html`
- `contact/index.html`
- `404.html`
- `evolther/index.html`

The generated shell currently appears in:

- `results/index.html`
- `results/*/index.html`
- copied result run pages under `results/*/run/index.html`, normalized by `tools/sync-results.mjs`

`careers/index.html` is a legacy redirect and is intentionally excluded from the shared shell checklist.

## Decision

Keep the site static and hand-authored for now. Do not introduce a framework, full static-site generator, or committed page-generation workflow for every route in this issue.

The operational source of truth is:

- `tools/sync-results.mjs` owns generated result pages and copied run-page shell normalization.
- Hand-authored pages remain manually edited, but they must follow this document and `node tools/check-site-shell.mjs`.
- `SITE_SHELL_VERSION` in `tools/sync-results.mjs` and hand-authored `styles.css` / `scripts.js` query strings must stay aligned.
- Per-page metadata stays local when it is intentionally page-specific, such as home copy, canonical paths, `robots` directives for experimental/noindex pages, and MathJax on result detail pages.

This is the smallest approach that prevents shell drift while preserving the repository's no-build deployment model.

## Rejected Alternatives

- Full framework: too much workflow and dependency weight for the current static site.
- Full static-site generator: useful later only if hand-authored duplication grows beyond checklist discipline.
- Generating all hand-authored pages now: would mix architecture churn with content risk and make review harder.
- Expanding `tools/sync-results.mjs` to own unrelated pages: that tool should remain focused on result publishing.

## Update Checklist

When changing global shell behavior:

1. Update hand-authored pages that participate in the shared shell.
2. Update `tools/sync-results.mjs` when generated results or copied run pages need the same shell change.
3. Keep the shared shell version aligned across HTML and `SITE_SHELL_VERSION`.
4. Preserve documented exceptions:
   - Home may omit the header brand because the wordmark is the primary hero.
   - Home and `404.html` may omit the footer.
   - `evolther/` and copied run pages may include `robots` noindex directives and may omit canonical/social metadata.
   - Result detail pages may include MathJax.
5. Run the shell checker before opening a PR.
6. Preview representative routes with a static server.

## Verification

Run:

```bash
node tools/check-site-shell.mjs
node tools/sync-results.mjs
git diff --check
python3 -m http.server 4173
```

Use `python3 -m http.server 4173` for the simplest static preview. Use `node tools/preview.mjs` when validating GitHub Pages-style 404 fallback behavior for unknown routes such as `/domains`; Python's built-in server shows its own error page for those routes. If port `4173` is already in use, run `node tools/preview.mjs 4174`.

Then spot-check:

- `/`
- `/company/`
- `/contact/`
- `/results/`
- one generated result page
- one copied run page
- `/404.html`
- `/evolther/`

## Follow-Ups

Split future work into focused issues if this checklist is no longer enough:

- Extract shared shell primitives into a small helper used by generated pages.
- Normalize metadata and canonical handling across hand-authored pages.
- Strengthen result generator shell integration.
- Add a stricter preview and visual QA checklist.
- Reconsider a static-site generator only if the manual/static model becomes a real maintenance bottleneck.
